# Usar este prompt con Claude para construir el proyecto desde cero

================================================================================
QUÉ ES EL PROYECTO
================================================================================

Un portal web de eventos musicales culturales (música latinoamericana en Berlín).
Permite a artistas publicar eventos, a usuarios explorar y filtrar contenido por
género y fecha, y a administradores gestionar usuarios y la plataforma.

Tres tipos de usuario:
- USER (default): puede explorar eventos y artistas
- ARTIST: puede crear y gestionar artistas y eventos propios
- ADMIN: acceso completo a toda la plataforma

Tres estados de usuario:
- PENDING: recién registrado, espera aprobación del admin
- ACTIVE: aprobado, puede operar normalmente
- BLOCKED: suspendido, sin acceso a funcionalidades

Tres idiomas: español (default), inglés, alemán.

================================================================================
STACK TECNOLÓGICO
================================================================================

- Next.js 15 con App Router (TypeScript)
- Better Auth — autenticación self-hosted, sin webhooks, escribe directo en DB
- PostgreSQL en Neon (serverless, free tier) + Prisma 6 como ORM
- next-intl — internacionalización (ES, EN, DE)
- Tailwind CSS 4 + shadcn/ui — estilos y componentes
- Cloudinary + next-cloudinary — almacenamiento y transformación de imágenes
- React Hook Form + Zod — formularios y validación
- Resend — emails transaccionales
- Trigger.dev — jobs asincrónicos (envío de emails en background)
- Vitest + Testing Library — tests unitarios y de componentes

================================================================================
AUTENTICACIÓN — BETTER AUTH
================================================================================

Better Auth es self-hosted: los usuarios se guardan directamente en tu
PostgreSQL (tabla `user`). No hay webhooks, no hay sincronización externa,
no hay dependencias de terceros con costo.

--- src/lib/auth.ts ---

import { betterAuth } from "better-auth"
import { prismaAdapter } from "better-auth/adapters/prisma"
import { admin } from "better-auth/plugins"
import { prisma } from "./prisma"
import { triggerWelcomeEmail } from "./trigger"

export const auth = betterAuth({
  database: prismaAdapter(prisma, { provider: "postgresql" }),
  emailAndPassword: { enabled: true },
  socialProviders: {
    google: {
      clientId: process.env.GOOGLE_CLIENT_ID!,
      clientSecret: process.env.GOOGLE_CLIENT_SECRET!,
    },
  },
  plugins: [admin()],
  session: {
    expiresIn: 60 * 60 * 24 * 7,   // 7 días
    updateAge: 60 * 60 * 24,        // refresh diario
  },
  hooks: {
    after: [
      {
        // Después del registro: crear UserProfile PENDING + email de bienvenida
        matcher: (context) => context.path === "/sign-up/email",
        handler: async (context) => {
          const user = context.context.newSession?.user
          if (user) {
            await prisma.userProfile.create({
              data: { userId: user.id, status: "PENDING" },
            })
            // Disparar email en background con Trigger.dev (no bloquea el response)
            await triggerWelcomeEmail({ email: user.email, name: user.name })
          }
        },
      },
    ],
  },
})

HELPERS DE AUTENTICACIÓN (src/services/auth.ts)
================================================================================

Toda la lógica de permisos vive aquí. Las API routes y los Server Components
importan estos helpers — nunca duplicar lógica de rol/status en otro lugar.

import { auth } from "@/lib/auth"
import { headers } from "next/headers"
import { redirect } from "next/navigation"
import { prisma } from "@/lib/prisma"

export async function getSession() {
  return auth.api.getSession({ headers: await headers() })
}

export async function getCurrentUser() {
  const session = await getSession()
  return session?.user ?? null
}

export async function requireAuth(locale = "es") {
  const user = await getCurrentUser()
  if (!user) redirect(`/${locale}/sign-in`)
  return user
}

export async function requireActive(locale = "es") {
  const user = await requireAuth(locale)
  const profile = await prisma.userProfile.findUnique({
    where: { userId: user.id },
  })
  if (profile?.status === "PENDING") redirect(`/${locale}/user-pending`)
  if (profile?.status === "BLOCKED") redirect(`/${locale}/user-blocked`)
  return { user, profile }
}

export async function requireRole(role: "ADMIN" | "ARTIST", locale = "es") {
  const { user } = await requireActive(locale)
  if (user.role !== role && user.role !== "ADMIN") {
    redirect(`/${locale}/dashboard`)
  }
  return user
}

export async function canEditArtist(artistId: string): Promise<boolean> {
  const user = await getCurrentUser()
  if (!user) return false
  if (user.role === "ADMIN") return true
  const artist = await prisma.artist.findUnique({ where: { id: artistId } })
  return artist?.userId === user.id
}

export async function canEditEvent(eventId: string): Promise<boolean> {
  const user = await getCurrentUser()
  if (!user) return false
  if (user.role === "ADMIN") return true
  const event = await prisma.event.findUnique({ where: { id: eventId } })
  return event?.createdById === user.id
}

================================================================================
MIDDLEWARE (src/middleware.ts)
================================================================================

Combinar next-intl con verificación de sesión de Better Auth.

import { betterFetch } from "@better-fetch/fetch"
import { NextResponse, type NextRequest } from "next/server"
import createIntlMiddleware from "next-intl/middleware"
import { routing } from "./i18n/routing"
import type { Session } from "better-auth/types"

const intlMiddleware = createIntlMiddleware(routing)

const PROTECTED_PATHS = ["/dashboard"]
const ADMIN_PATHS = ["/admin"]

export async function middleware(request: NextRequest) {
  const response = intlMiddleware(request)

  const { pathname } = request.nextUrl
  const locale = pathname.split("/")[1] || "es"
  const pathWithoutLocale = "/" + pathname.split("/").slice(2).join("/")

  const isProtected = PROTECTED_PATHS.some((p) => pathWithoutLocale.startsWith(p))
  const isAdmin = ADMIN_PATHS.some((p) => pathWithoutLocale.startsWith(p))

  if (isProtected || isAdmin) {
    const { data: session } = await betterFetch<Session>(
      "/api/auth/get-session",
      {
        baseURL: request.nextUrl.origin,
        headers: { cookie: request.headers.get("cookie") ?? "" },
      }
    )

    if (!session) {
      return NextResponse.redirect(new URL(`/${locale}/sign-in`, request.url))
    }

    if (isAdmin && session.user.role !== "ADMIN") {
      return NextResponse.redirect(new URL(`/${locale}/dashboard`, request.url))
    }
  }

  return response
}

export const config = {
  matcher: ["/((?!api|_next|.*\\..*).*)"],
}

// Eliminar acentos y generar slug limpio
export function slugify(text: string): string {
  return text
    .toLowerCase()
    .normalize("NFD")
    .replace(/[\u0300-\u036f]/g, "")
    .replace(/[^a-z0-9\s-]/g, "")
    .trim()
    .replace(/\s+/g, "-")
}

// Generar slug único verificando colisiones en DB
export async function generateUniqueSlug(
  base: string,
  model: "event" | "artist"
): Promise<string> {
  const slug = slugify(base)
  const existing = await prisma[model].findUnique({ where: { slug } })
  if (!existing) return slug
  const similar = await prisma[model].findMany({
    where: { slug: { startsWith: slug } },
    select: { slug: true },
  })
  const max = similar.reduce((n, { slug: s }) => {
    const match = s.match(/-(\d+)$/)
    return match ? Math.max(n, parseInt(match[1])) : n
  }, 1)
  return `${slug}-${max + 1}`
}

// Formatear fecha según locale activo
export function formatDate(date: Date, locale: string): string {
  const localeMap = { es: "es-ES", en: "en-US", de: "de-DE" }
  return new Intl.DateTimeFormat(localeMap[locale] ?? "es-ES", {
    weekday: "long",
    year: "numeric",
    month: "long",
    day: "numeric",
  }).format(date)
}

export async function POST(request: Request) {
  // 1. Verificar sesión

  if (!user) return NextResponse.json({ error: "Unauthorized", code: "UNAUTHORIZED" }, { status: 401 })

  // 2. Verificar permiso específico usando helpers de services/auth.ts
  // 3. Validar body con Zod
  const body = await request.json()
  const result = schema.safeParse(body)
  if (!result.success) {
    return NextResponse.json({ error: "Validation error", code: "VALIDATION_ERROR", issues: result.error.issues }, { status: 400 })
  }

  // 4. Llamar al service correspondiente
  // 5. Retornar respuesta estructurada

  return NextResponse.json({ data: result })
}

================================================================================
REGLAS DE ARQUITECTURA — EL AGENTE DEBE SEGUIRLAS SIN EXCEPCIÓN
================================================================================

1. SERVER COMPONENTS POR DEFECTO.
   Agregar "use client" solo cuando se necesiten: hooks de estado (useState,
   useEffect), eventos del DOM, o APIs del browser. Todo fetch de datos inicial
   va en Server Components.

2. NO useEffect PARA FETCHING.
   useEffect solo para efectos de UI puros (scroll, animaciones, DOM). Los datos
   se obtienen en Server Components o Server Actions.

3. TRANSACCIONES EN OPERACIONES COMPUESTAS.
   Crear evento con fechas e imágenes siempre dentro de prisma.$transaction().
   Si falla cualquier parte, se revierte todo.

4. UN SOLO PUNTO DE VERDAD PARA PERMISOS.
   Toda la lógica de rol/status vive en src/services/auth.ts. Las API routes y
   los Server Components importan esos helpers. Nunca duplicar guards de permisos.

5. SOFT DELETE OBLIGATORIO PARA EVENTOS.
   isDeleted + deletedAt. Todos los queries de listado excluyen { isDeleted: true }.
   Hard delete solo para ADMIN y borra también las imágenes de Cloudinary primero.

6. TIPOS DE DOMINIO SEPARADOS DE PRISMA.
   Los services mapean los resultados de Prisma a interfaces de dominio propias
   antes de retornarlos. Los componentes nunca importan tipos de Prisma directamente.

7. SLUGS ÚNICOS CON COLISIÓN RESUELTA.
   Usar generateUniqueSlug() siempre al crear artistas o eventos. Nunca asumir
   que el slug es único sin verificar en la DB.

8. IMÁGENES: SIEMPRE GUARDAR publicId.
   Al eliminar cualquier entidad (artista, evento), eliminar primero sus imágenes
   en Cloudinary usando el publicId, luego eliminar en la DB.

9. CACHE CON unstable_cache.
   Queries de lectura frecuente (listados de eventos y artistas) deben estar
   cacheadas. Invalidar con revalidateTag() al mutar datos.

10. EMAILS SIEMPRE EN BACKGROUND.
    Nunca enviar emails de forma síncrona en un request. Usar Trigger.dev para
    disparar el job y responder inmediatamente al usuario.

11. ENV.TS COMO ÚNICA FUENTE DE ENV VARS.
    Importar siempre desde src/lib/env.ts, nunca acceder a process.env directamente
    en componentes o servicios.

12. NO CREAR ABSTRACCIONES PARA UN SOLO USO.
    No agregar features no pedidos. No agregar comentarios obvios. No crear helpers
    especulativos. La complejidad justa es la que el problema requiere.
